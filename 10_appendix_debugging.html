🔧 APPENDIX: DEBUGGING TECHNIQUES

Complete Debugging Toolkit for Embedded Linux Development

SECTION 1: USER-SPACE DEBUGGING

strace — System Call Tracer

The #1 tool for understanding what a user-space program does:

# Trace all syscalls of a command
strace ls -la

# Trace your test app
strace ./test_app

# Focus on specific syscall types
strace -e trace=open,read,write,close ./test_app
strace -e trace=ioctl ./test_app
strace -e trace=memory ./test_app       # mmap, munmap, etc.

# With timestamps
strace -t ./test_app                    # HH:MM:SS
strace -T ./test_app                    # Time spent in each syscall

# Follow forked children
strace -f ./program_that_forks

# Write output to file (useful for long traces)
strace -o trace.log ./test_app

# Count and summarize syscalls
strace -c ./test_app

Example output analysis:

open("/dev/chardev0", O_RDWR) = 3       ← fd=3, success
write(3, "Hello", 5) = 5                ← 5 bytes written, OK
read(3, "Hello", 100) = 5               ← Read back 5 bytes
ioctl(3, 0xC0044301, 0x7ffe...) = 0     ← IOCTL success
close(3) = 0                             ← Clean close

write(3, "test", 4) = -1 EBADF          ← fd 3 already closed!
open("/dev/missing", O_RDWR) = -1 ENOENT ← Device doesn't exist

GDB — The GNU Debugger

# Compile with debug info
gcc -g -O0 -o test_app test_app.c

# Start debugging
gdb ./test_app

# Essential GDB commands:
(gdb) break main              # Breakpoint at main
(gdb) break test_app.c:42     # Breakpoint at line 42
(gdb) run                     # Start program
(gdb) next                    # Step over (execute line, don't enter functions)
(gdb) step                    # Step into function
(gdb) print variable          # Print variable value
(gdb) print *ptr              # Dereference pointer
(gdb) print sizeof(struct my_struct)
(gdb) x/16xb ptr              # Examine memory: 16 bytes, hex
(gdb) bt                      # Backtrace (call stack)
(gdb) info threads            # List threads
(gdb) thread 2                # Switch to thread 2
(gdb) watch variable          # Break when variable changes
(gdb) continue                # Resume execution
(gdb) quit

Valgrind — Memory Error Detector

# Detect memory errors
valgrind --leak-check=full ./test_app

# Common outputs:
# "Invalid read of size 4" → Use-after-free or out-of-bounds
# "Conditional jump depends on uninitialised value" → Using uninitialized memory
# "definitely lost: 48 bytes" → Memory leak

# Thread error detector
valgrind --tool=helgrind ./threaded_app
# Detects: data races, lock ordering issues, mutex misuse

Address Sanitizer (ASan) — Compile-Time Instrumentation

# Compile with ASan (better performance than valgrind)
gcc -fsanitize=address -g -o test_app test_app.c

# Detects:
# - Buffer overflow (stack and heap)
# - Use-after-free
# - Double-free
# - Memory leaks

# Thread Sanitizer
gcc -fsanitize=thread -g -o test_app test_app.c thread_pool.c -pthread
# Detects: data races between threads

SECTION 2: KERNEL-SPACE DEBUGGING

dmesg — Kernel Message Buffer

# View kernel messages
dmesg                        # All messages
dmesg | tail -20             # Last 20 lines
dmesg -w                     # Watch in real-time (like tail -f)
dmesg --level=err            # Only errors
dmesg -T                     # Human-readable timestamps
dmesg -c                     # Read AND clear buffer

# Filter by your module
dmesg | grep my_module

# Log levels (0=highest priority):
# KERN_EMERG   (0) - System unusable
# KERN_ALERT   (1) - Action required immediately
# KERN_CRIT    (2) - Critical
# KERN_ERR     (3) - Error
# KERN_WARNING (4) - Warning
# KERN_NOTICE  (5) - Normal but significant
# KERN_INFO    (6) - Informational
# KERN_DEBUG   (7) - Debug messages

# Set console log level (show level 7 and above on console)
echo 7 > /proc/sys/kernel/printk

Dynamic Debug (dev_dbg at runtime)

# Enable ALL debug messages from your module
echo 'module temp_sensor +p' > /sys/kernel/debug/dynamic_debug/control

# Enable for specific file
echo 'file chardev.c +p' > /sys/kernel/debug/dynamic_debug/control

# Enable for specific function
echo 'func chardev_read +p' > /sys/kernel/debug/dynamic_debug/control

# Add function name and line number to output
echo 'module temp_sensor +fpl' > /sys/kernel/debug/dynamic_debug/control

# List all available debug sites
cat /sys/kernel/debug/dynamic_debug/control | grep temp_sensor

# Disable
echo 'module temp_sensor -p' > /sys/kernel/debug/dynamic_debug/control

/proc and /sys Inspection

# ==== MODULE INFO ====
cat /proc/modules                         # All loaded modules
lsmod                                     # Same, formatted
modinfo my_module.ko                      # Module metadata

# ==== DEVICE INFORMATION ====
cat /proc/devices                          # Major numbers and drivers
ls -la /dev/chardev*                       # Device nodes
cat /sys/class/chardev_class/chardev0/dev  # Major:minor

# ==== DRIVER-DEVICE BINDING ====
ls /sys/bus/platform/drivers/mydevice/     # Platform driver
ls /sys/bus/i2c/drivers/temp_sensor/       # I2C driver
cat /sys/bus/i2c/devices/1-0048/name       # I2C device name

# ==== I2C DEBUGGING ====
i2cdetect -l                               # List I2C buses
i2cdetect -y 1                             # Scan bus 1
i2cget -y 1 0x48 0x00 w                   # Read register
i2cset -y 1 0x48 0x01 0x6000 w            # Write register
i2cdump -y 1 0x48                          # Dump all registers

# ==== MEMORY INFO ====
cat /proc/meminfo                          # System memory
cat /proc/slabinfo                         # Slab allocator stats
cat /proc/buddyinfo                        # Free page counts

# ==== INTERRUPT INFO ====
cat /proc/interrupts                       # IRQ counts per CPU
watch -n1 'cat /proc/interrupts | head -20' # Watch interrupts

# ==== GPIO DEBUGGING ====
cat /sys/kernel/debug/gpio                 # GPIO state (if debugfs mounted)

ftrace — Kernel Function Tracer

# Mount debugfs if not mounted
mount -t debugfs none /sys/kernel/debug

# Trace function calls
cd /sys/kernel/debug/tracing

# List available tracers
cat available_tracers

# Trace kernel function graph (who calls what)
echo function_graph > current_tracer
echo 1 > tracing_on
# ... trigger your driver ...
echo 0 > tracing_on
cat trace

# Trace only your module's functions
echo ':mod:my_module' > set_ftrace_filter
echo function > current_tracer
echo 1 > tracing_on
insmod my_module.ko
echo 0 > tracing_on
cat trace

# Reset
echo nop > current_tracer
echo > set_ftrace_filter

Kernel Oops — Reading Crash Outputs

When your module crashes, you'll see something like:

BUG: unable to handle page fault at ffffffffc0000000
Oops: 0002 [#1] SMP
CPU: 0 PID: 1234 Comm: test_app Tainted: G    OE    5.15.0
RIP: 0010:chardev_write+0x42/0x100 [chardev]     ← CRASH LOCATION
RSP: 0018:ffffc900001234
RAX: 0000000000000000 RBX: ffff888012345678       ← Register dump
...
Call Trace:
 vfs_write+0xc5/0x200                              ← Call stack
 ksys_write+0x5f/0xe0
 do_syscall_64+0x33/0x80
 entry_SYSCALL_64_after_hwframe+0x44/0xae

HOW TO READ:
1. "RIP: 0010:chardev_write+0x42/0x100 [chardev]"
   → Crashed in chardev_write(), 0x42 bytes into the function
   → Module: [chardev]

2. Find exact line:
   $ addr2line -e chardev.ko 0x42
   or
   $ gdb chardev.ko
   (gdb) list *(chardev_write+0x42)

3. "page fault at ffffffffc0000000"
   → NULL pointer dereference (or near-NULL)
   → Check: did you initialize the pointer?

4. "Oops: 0002" 
   → Bit 0: 0=no page found, 1=protection fault
   → Bit 1: 0=read, 1=write (0002 = write to unmapped page)

SECTION 3: DEBUGGING WORKFLOW

When Your Module Won’t Load

PROBLEM: insmod returns -1

CHECK:
1. dmesg | tail        → See YOUR init function's error message
2. Is it a dependency?  → modprobe (not insmod) resolves deps
3. Version mismatch?   → modinfo my_module.ko | grep vermagic
                          uname -r
                          Must match!
4. Missing symbol?     → dmesg: "Unknown symbol kmalloc_..."
                          → Need kernel headers: apt install linux-headers-$(uname -r)

When Your Driver Crashes

1. Enable debug: echo 8 > /proc/sys/kernel/printk
2. Add pr_info at every stage of your function
3. Check dmesg for the Oops
4. Use addr2line to find exact line
5. Common causes:
   - NULL dereference → check all pointer returns
   - copy_to_user with bad pointer → check user args
   - Sleeping in atomic context → no mutex in IRQ
   - Missing lock → race condition

When I2C Communication Fails

1. Is the device present?
   i2cdetect -y 1
   → UU means driver bound, -- means not found

2. Is the bus working?
   ls /dev/i2c-*
   If empty: I2C adapter driver not loaded

3. Manual test:
   i2cget -y 1 0x48 0x00    (read one byte)
   Error? → Check wiring, pull-ups, address

4. In your driver:
   Add dev_info in probe for every i2c_smbus call result
   Check return values!

SECTION 4: PERFORMANCE DEBUGGING

perf — The Swiss Army Knife

# Record CPU profile of your test
sudo perf record -g ./test_app
sudo perf report

# System-wide profiling
sudo perf top

# Count cache misses
sudo perf stat -e cache-misses ./test_app

# Trace events
sudo perf trace -e 'syscalls:sys_enter_ioctl' ./test_app

Kernel Config Options for Debugging

CONFIG_DEBUG_KERNEL=y         # Master debug switch
CONFIG_KASAN=y                # Kernel Address Sanitizer
CONFIG_KMEMLEAK=y             # Memory leak detector
CONFIG_PROVE_LOCKING=y        # Lockdep: detect lock issues
CONFIG_DEBUG_ATOMIC_SLEEP=y   # Detect sleeping in atomic
CONFIG_DEBUG_INFO=y           # Debug symbols in vmlinux
CONFIG_DYNAMIC_DEBUG=y        # Dynamic pr_debug/dev_dbg
CONFIG_FTRACE=y               # Function tracing

# Check current config:
cat /boot/config-$(uname -r) | grep CONFIG_KASAN
# or
zcat /proc/config.gz | grep KASAN  # if available

Problem Tool Command
“What syscalls does my app make?” strace strace -e trace=ioctl ./app
“Where does my app crash?” GDB gdb ./app then run, bt

“Data race in threads?” TSan gcc -fsanitize=thread
“Module won’t load?” dmesg dmesg \| tail -10
“Module crashes kernel?” Oops + addr2line Read dmesg Oops
“Driver doesn’t probe?” sysfs ls /sys/bus/*/drivers/mydrv/
“I2C device not responding?” i2cdetect i2cdetect -y 1
“What functions are slow?” ftrace/perf perf record -g
“Lock ordering bug?” lockdep Enable CONFIG_PROVE_LOCKING
