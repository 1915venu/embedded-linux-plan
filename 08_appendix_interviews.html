📚 COMPREHENSIVE INTERVIEW QUESTION BANK

150+ Questions Categorized by Topic — with Model Answers for 0-2 YOE

CATEGORY 1: C PROGRAMMING FUNDAMENTALS (25 Questions)

Memory & Pointers

Q1: What is the difference between stack and heap? How does this relate to kernel programming?

Stack: auto-allocated, LIFO, ~8MB user / 8-16KB kernel. Heap: manual via malloc, unlimited. In kernel: NO malloc — use kmalloc (slab allocator). Kernel stack is extremely limited, so never declare large local arrays.

Q2: What is a memory leak? How do you detect one in kernel space?

Memory allocated but never freed. User: valgrind, ASan. Kernel: CONFIG_DEBUG_KMEMLEAK — kmemleak scans memory for unreferenced allocations. Also: cat /proc/slab_allocators.

Q3: What is the container_of macro? Give a kernel use case.

Given pointer to member, returns pointer to enclosing struct: container_of(ptr, type, member). Used in kernel linked lists: you embed struct list_head in your struct, then recover your struct from the list node pointer. Also used in cdev → your device struct in driver’s open().

Q4: Explain volatile keyword and when to use it.

Tells compiler: don’t optimize accesses, always read from memory. Use for: memory-mapped hardware registers, variables shared with ISR, signal handlers. In kernel: I/O accessors (readl()/writel()) include barriers, so volatile is less common but volatile is needed for direct MMIO pointers.

Q5: What is restrict keyword?

Promise to compiler that a pointer is the only way to access that memory. Enables better optimization. Used in: memcpy(void * restrict dest, const void * restrict src, size_t n) — guarantees no overlap.

Q6: Explain the difference between sizeof and strlen.

sizeof: compile-time operator, gives size of type/variable in bytes (includes padding, null terminator). strlen: runtime function, counts chars until \0. sizeof("hello") = 6 (includes \0), strlen("hello") = 5.

Q7: What is a union? How is it used in embedded?

Union members share same memory location. Size = largest member. Use: type punning, protocol parsing, register overlays. c union reg_access { uint32_t raw; struct { uint8_t byte0, byte1, byte2, byte3; } bytes; };

Q8: What is the offsetof macro?

Returns byte offset of member within struct: offsetof(struct_type, member). Foundation of container_of. Defined as ((size_t)&((type*)0)->member).

Q9: Explain static keyword in all its uses in C.

  1. static local variable: persists across function calls (stored in DATA/BSS, not stack).
  2. static global/function: limits linkage to current file (internal linkage — like “private” in OOP).
  3. In kernel: static functions/variables are file-scope — standard practice to avoid symbol collisions between modules.

Q10: What is inline and does the compiler always inline?

Hint to compiler to expand function at call site (eliminates call overhead). Compiler may ignore hint. In kernel: __always_inline forces inlining. inline helps for small, frequently-called functions. Trade-off: code size vs call overhead.

Bitwise

Q11: How do you atomically set a bit in a shared variable?

Userspace: __atomic_or_fetch(&var, BIT(n), __ATOMIC_SEQ_CST) or atomic_fetch_or(). Kernel: set_bit(n, &var) which uses architecture-specific atomic instructions (LOCK prefix on x86, LDREX/STREX on ARM).

Q12: Write code to reverse the bits of a 32-bit integer.

c uint32_t reverse_bits(uint32_t n) { n = ((n >> 1) & 0x55555555) | ((n & 0x55555555) << 1); n = ((n >> 2) & 0x33333333) | ((n & 0x33333333) << 2); n = ((n >> 4) & 0x0F0F0F0F) | ((n & 0x0F0F0F0F) << 4); n = ((n >> 8) & 0x00FF00FF) | ((n & 0x00FF00FF) << 8); n = (n >> 16) | (n << 16); return n; }

Q13: What is two’s complement representation?

System to represent signed integers. Positive: same as unsigned. Negative: flip all bits + 1. So -1 = 0xFFFFFFFF. Advantage: single representation of zero, addition works for both signed/unsigned.

Threading

Q14: What is a futex and how does pthread_mutex use it?

Fast Userspace Mutex. Uncontested lock/unlock: atomic CAS in userspace (no syscall!). Only when contested: futex() syscall to sleep/wake. Huge performance win — most locks are uncontested.

Q15: What is the ABA problem in lock-free programming?

Thread reads value A, gets preempted. Another thread changes A→B→A. First thread’s CAS succeeds (value is A again) but state has changed. Solution: add version counter alongside the value.

CATEGORY 2: LINUX KERNEL FUNDAMENTALS (25 Questions)

Q16: What is the difference between kernel space and user space?

Kernel: Ring 0, full HW access, all memory, single shared address space. User: Ring 3, restricted, isolated per-process virtual address space. Crossing boundary = syscall = context switch (save/restore registers, switch stack).

Q17: What happens during a system call? Trace it step by step.

  1. User calls library function (write()). 2. glibc wrapper loads syscall# into rax. 3. Args in rdi, rsi, rdx, etc. 4. SYSCALL instruction. 5. CPU switches to Ring 0. 6. Kernel saves user registers. 7. Look up handler in sys_call_table. 8. Execute handler (ksys_write). 9. Return value in rax. 10. SYSRET back to user mode.

Q18: What is kernel preemption?

Kernel can be preempted to run higher-priority task. CONFIG_PREEMPT_NONE: never preempt kernel (server). CONFIG_PREEMPT: preempt almost anywhere except critical sections (desktop). CONFIG_PREEMPT_RT: full preemption (real-time).

Q19: Explain the Linux kernel module lifecycle.

  1. insmod module.ko → load binary into kernel memory. 2. Resolve symbols (link to kernel exports like printk). 3. Call module_init() function. 4. If returns 0 → module active, appears in lsmod. 5. rmmod module → Call module_exit() function. 6. Free module memory.

Q20: What is EXPORT_SYMBOL?

Makes a kernel function/variable available to other modules. Without it, a function is only usable within its own module. EXPORT_SYMBOL_GPL limits to GPL-licensed modules. Exported symbols appear in /proc/kallsyms.

Q21: What happens if module_init returns non-zero?

Module loading fails. Error code returned to insmod. Module_exit is NOT called! You MUST clean up any partially-allocated resources before returning the error (this is why goto cleanup pattern exists).

Q22: What is printk vs pr_info?

printk(KERN_INFO "msg") is the raw function. pr_info("msg") is a convenience macro that adds KERN_INFO level. pr_err, pr_warn, pr_debug for other levels. dev_info(dev, "msg") adds device name prefix — preferred in drivers.

Q23: When can you sleep in kernel code?

Only in process context (system calls, work queues, kernel threads). NEVER in: interrupt handlers (top half), softirqs, tasklets, or while holding spinlocks. Use might_sleep() debug assertion to verify.

Q24: What are the kernel log levels?

0=EMERG, 1=ALERT, 2=CRIT, 3=ERR, 4=WARNING, 5=NOTICE, 6=INFO, 7=DEBUG. Messages below console_loglevel appear on console. Default: 4 (KERN_WARNING and above).

Q25: What is the __init and __initdata annotation?

__init: function only needed during initialization. Memory freed after init. __initdata: same for data. Saves memory in running kernel. If module is built-in, this memory is reclaimed. If loadable module, __init memory freed after successful insmod.

CATEGORY 3: MEMORY MANAGEMENT (20 Questions)

Q26: kmalloc vs vmalloc vs kzalloc — when to use each?

kmalloc: physically contiguous, fast, < 128KB, for DMA-able buffers. vmalloc: virtually contiguous only, large allocations, slower, can’t DMA. kzalloc: kmalloc + zero-initialized. Rule: use kzalloc for structures, kmalloc for DMA, vmalloc for large non-DMA.

Q27: What is GFP_KERNEL vs GFP_ATOMIC?

GFP_KERNEL: normal allocation, can sleep (block) waiting for memory. Use in process context. GFP_ATOMIC: cannot sleep, for interrupt context or spinlock-held context. May fail more often because can’t reclaim memory.

Q28: What is the slab allocator?

Caches of commonly-sized objects. Reduces fragmentation vs raw page allocation. Kernel uses SLUB (default), SLAB, or SLOB. kmem_cache_create() for driver-specific caches. View: cat /proc/slabinfo.

Q29: Explain the page allocator (buddy system).

Allocates physical pages in powers of 2 (1, 2, 4, 8… pages). Fast allocation/freeing. Handles fragmentation by splitting and merging buddies. alloc_pages() returns struct page*.

Q30: What is devm_kzalloc and why use it?

“Managed” allocation — automatically freed when device is removed or probe fails. No need to manually free in remove() or error paths. Almost always preferred in modern drivers. devm_* family includes devm_ioremap, devm_request_irq, etc.

Q31: What is DMA-able memory? How do you allocate it?

Memory accessible by hardware DMA engines. Must be physically contiguous. Use dma_alloc_coherent() for consistent DMA, dma_map_single() for streaming DMA. Never use vmalloc for DMA.

Q32: What is the difference between ioremap and mmap?

ioremap(): kernel function, maps physical I/O addresses into kernel virtual space. Returns void __iomem*. Use to access hardware registers from driver. mmap(): user syscall, maps files/memory into user virtual space.

Q33: What is __iomem annotation?

Sparse annotation marking pointers to I/O memory (not regular RAM). Must use readl()/writel() to access, not direct dereference. Sparse checker warns if you dereference __iomem directly.

Q34: What is copy_to_user / copy_from_user?

Safe data transfer between kernel and user address spaces. Validates user pointer, handles page faults, returns number of bytes NOT copied (0=success). Using memcpy with user pointers would crash kernel on bad addresses.

Q35: What is the high memory problem (32-bit)?

32-bit kernel: ~1GB virtual address space. If system has >1GB RAM, some physical pages can’t be permanently mapped. These are “high memory” pages, accessed via temporary mappings (kmap()/kunmap()). Not an issue on 64-bit.

Q36: What is MMIO vs Port I/O?

MMIO: hardware registers mapped to memory addresses, accessed with readl/writel. Standard on ARM/modern x86. Port I/O: separate I/O address space on x86, accessed with inb/outb. Legacy x86 peripherals use port I/O.

Q37: Explain kernel virtual address space layout.

On 64-bit x86: User space 0x0-0x7FFF… (128TB). Kernel starts at 0xFFFF… Direct mapping of all physical RAM. vmalloc area. Module space. fixmap area.

Q38: What is a memory barrier and why needed?

CPUs/compilers reorder memory operations for performance. Barriers enforce ordering. smp_rmb(): read barrier. smp_wmb(): write barrier. smp_mb(): full barrier. I/O functions and locks include appropriate barriers.

Q39: What is NUMA-aware allocation?

Non-Uniform Memory Access: different CPU/memory node distances. kmalloc_node() allocates from specific NUMA node. Important for performance on multi-socket servers.

Q40: What is kernel OOM (Out of Memory) killer?

When system runs out of memory, OOM killer selects and terminates a process to free memory. Selection based on memory usage, oom_score_adj. Kernel drivers should handle allocation failures gracefully.

CATEGORY 4: DEVICE DRIVERS (30 Questions)

Q41: Character vs Block vs Network devices — explain each.

Character: byte-stream, sequential, no caching (serial ports, sensors). Block: random access, sector-based, kernel provides caching (disks, flash). Network: packet-based, special subsystem, no /dev entry (eth0, wlan0).

Q42: Walk me through what happens when user calls open("/dev/mydev").

  1. VFS resolves path to inode. 2. Inode has major/minor number. 3. Kernel looks up character device by major number → finds your struct cdev. 4. Gets file_operations from cdev. 5. Allocates struct file. 6. Calls your .open() handler, passing inode and file. 7. Returns fd to user.

Q43: What is file_operations structure?

Table of function pointers that defines how a character device responds to file operations: .open(), .read(), .write(), .release(), .unlocked_ioctl(), .llseek(), .mmap(), .poll(). This is effectively a vtable — polymorphism in C.

Q44: Explain struct file vs struct inode.

inode: represents a file on disk/filesystem. One per file. Contains permissions, size, device numbers. Persists as long as file exists. file: represents an open instance of a file. One per open() call. Contains position (f_pos), flags, private_data. Multiple file structs can point to same inode.

Q45: What is file->private_data and how do you use it?

void* in struct file for driver-private data. Set in .open(): filp->private_data = my_device_struct;. Read in .read(), .write(), .ioctl(). Allows one driver to handle multiple devices — private_data points to per-device state.

Q46: Why container_of in open() instead of global variable?

Multiple device instances. container_of(inode->i_cdev, struct my_dev, cdev) recovers your device struct from the cdev embedded in it. Global variable only works for single device.

Q47: Explain alloc_chrdev_region vs register_chrdev_region.

alloc_chrdev_region: kernel chooses major number dynamically. Preferred — avoids conflicts. register_chrdev_region: you specify major number. Static allocation — risk of conflict. Old API.

Q48: What is cdev_init + cdev_add?

cdev_init(&cdev, &fops): initialize cdev with your file_operations. cdev_add(&cdev, dev_t, count): register with kernel, making device callable. Must do both — init sets up fops, add makes it findable by VFS.

Q49: What is class_create + device_create?

class_create("my_class"): creates device class in /sys/class/my_class/. device_create(class, parent, dev_t, data, "mydev%d", minor): creates device entry → triggers udev → creates /dev/mydevN automatically. Without this, user must manually mknod.

Q50: Explain IOCTL — what it is and how to implement.

IOCTL (I/O Control): custom device commands beyond read/write. Define commands with macros: _IO(magic, nr) (no data), _IOR(magic, nr, type) (read from driver), _IOW(magic, nr, type) (write to driver), _IOWR (both). Handler validates magic with _IOC_TYPE(cmd), uses switch on command, copy_to/from_user for data.

Q51: What is the difference between mutex_lock and mutex_lock_interruptible?

mutex_lock: blocks until available, cannot be interrupted by signals. mutex_lock_interruptible: can be interrupted — returns -ERESTARTSYS if signal received. Use interruptible in drivers so Ctrl+C works when user process is blocked.

Q52: How do you handle concurrent access to your driver?

Use mutex for process-context operations (read/write/ioctl). Lock before accessing shared data, unlock after. Use mutex_lock_interruptible() for tasks that user should be able to cancel. For ISR+process interaction: spinlock + irqsave.

Q53: What is a device node and how is it created?

Special file in /dev with major:minor numbers. Created by: 1) device_create() + udev (automatic, preferred), 2) mknod /dev/mydev c MAJOR MINOR (manual), 3) devtmpfs (early boot).

Q54: What is the probe/remove lifecycle?

Modern Linux driver model. probe(): called when kernel matches device to driver (by compatible string or name). Initialize hardware, allocate resources, register interfaces. remove(): called on unload. Release resources. With devm_*, remove() can often be empty.

Q55: What is deferred probing (-EPROBE_DEFER)?

If probe() depends on resource not yet available (clock, regulator, GPIO controller), return -EPROBE_DEFER. Kernel retries later when more subsystems are initialized. Essential for handling driver initialization ordering.

Q56: What is sysfs and how do drivers use it?

Kernel object filesystem at /sys. Drivers create attributes using DEVICE_ATTR_RO/RW/WO macros. Each attribute → one file in sysfs. User reads/writes: cat /sys/class/myclass/mydev/temp. Preferred over ioctl for simple values.

Q57: Explain the top-half / bottom-half interrupt model.

Top half (ISR): runs in interrupt context, must be fast. Acknowledge interrupt, grab urgent data, schedule bottom half. Bottom half: runs in process context (can sleep). Work queue: schedule_work(). Tasklet: tasklet_schedule() (softirq context, can’t sleep). Threaded IRQ: devm_request_threaded_irq() — preferred modern approach.

Q58: What is a work queue?

Kernel mechanism to defer work to process context. INIT_WORK(&work, handler_func) + schedule_work(&work). Handler runs in kernel thread (can sleep, use mutex). Preferred bottom-half mechanism in modern drivers.

Q59: What is a threaded IRQ?

request_threaded_irq(irq, hardirq_handler, thread_handler, flags, ...). Hard IRQ: runs in interrupt context, quickly acknowledges. Thread handler: runs in dedicated kernel thread, can sleep, can use mutex. Modern preferred approach when bottom-half needs to sleep.

Q60: How does DMA work in Linux drivers?

  1. Allocate DMA-able buffer: dma_alloc_coherent() for consistent, or dma_map_single() for streaming. 2. Program DMA controller with physical address. 3. Start transfer. 4. Wait for completion (interrupt or polling). 5. Unmap/free buffer.

CATEGORY 5: DEVICE TREE (15 Questions)

Q61: What problem does Device Tree solve?

Before DT, ARM had board-specific C code in arch/arm/mach-*. Each board = kernel recompile. DT separates hardware description from code. Same kernel binary works on multiple boards — only DTB changes.

Q62: How does compatible string matching work?

Device node: compatible = "vendor,device-v2", "vendor,device-v1", "generic". Driver: of_match_table lists compatible strings. Kernel tries each device string until it finds a match in some driver’s table. First entry = most specific.

Q63: What is #address-cells and #size-cells?

Defines format of reg property in child nodes. #address-cells = <1>: 32-bit addresses. #size-cells = <0>: no size (I2C addresses). Example: parent has #size-cells = <0>, child reg = <0x48> = just address.

Q64: How do you read DT properties in driver code?

of_property_read_u32(node, "my-prop", &val), of_property_read_string(node, "name", &str), of_property_read_bool(node, "flag") (presence = true), of_property_read_u32_array(node, "arr", arr, count).

Q65: What is a DT overlay?

Modifies base DT at runtime without replacing it. Used for: hot-pluggable hardware (capes, HATs), enabling/disabling peripherals. Syntax: fragment@0 { target = <&node>; __overlay__ { ... }; };.

Q66: What is of_device_get_match_data?

Returns .data from matching of_device_id entry. Use to pass version-specific configuration. No need for runtime string comparison — kernel already matched.

Q67: What is the status property?

"okay": device enabled. "disabled": device present on SoC but not populated/used on this board. Kernel skips disabled nodes. Overlays can change disabled→okay.

Q68: What is ranges property?

Address translation between parent and child address spaces. ranges; (empty): 1:1 mapping. ranges = <child_addr parent_addr size>: explicit translation. Without ranges, child addresses are in their own space.

Q69: How are interrupts described in DT?

interrupt-parent = <&intc>: which interrupt controller. interrupts = <TYPE NUM FLAGS>: interrupt specification. For ARM GIC: interrupts = <GIC_SPI 45 IRQ_TYPE_LEVEL_HIGH>.

Q70: What is a phandle?

Handle to reference another node. clocks = <&clock_controller 5>: reference to clock_controller node, clock ID 5. &label syntax creates phandle.

CATEGORY 6: I2C/SPI PROTOCOLS (15 Questions)

Q71: Describe the I2C protocol.

Two wires: SDA (data), SCL (clock). Open-drain with pull-ups. Master generates clock. 7-bit addressing (10-bit optional). START → Address(7)+R/W(1) → ACK → Data(8) → ACK → … → STOP. Speeds: 100kHz (standard), 400kHz (fast), 1MHz (fast+), 3.4MHz (high-speed).

Q72: I2C vs SPI — when to use each?

I2C: 2 wires, multi-master, 7-bit addressing, slower (100-400kHz). Good for: sensors, EEPROMs, many slow devices. SPI: 4+ wires (MOSI, MISO, SCK, CS), chip-select per device, faster (MHz-100MHz), full duplex. Good for: flash, displays, ADCs.

Q73: What is the difference between i2c_client and i2c_adapter?

i2c_adapter: the I2C bus/controller hardware. Provides master_xfer(). One per I2C bus. i2c_client: specific device on bus. Contains address, adapter reference. Your driver gets i2c_client in probe().

Q74: Explain i2c_smbus_read_word_swapped.

I2C sends MSB first. On little-endian host (x86, ARM-LE), bytes need swapping. _swapped variant does this automatically. Without it: manual be16_to_cpu() needed.

Q75: How does your I2C driver handle errors?

Check every i2c_smbus_* return value. Negative = error. Common errors: -ENODEV (no ACK from device), -ETIMEDOUT (bus stuck), -EIO (general I/O error). Propagate errors to caller. Consider retry for transient errors.

Q76: What is SMBus and how does it relate to I2C?

SMBus is a subset of I2C for system management. Defines specific transaction types (byte data, word data, block). i2c_smbus_* functions implement SMBus protocol. Most I2C devices support SMBus transactions.

Q77: How do you instantiate an I2C device in Linux?

  1. Device Tree: temp_sensor@48 { compatible = "ti,tmp102"; reg = <0x48>; };
  2. ACPI tables (x86)
  3. Platform code: i2c_register_board_info()
  4. Sysfs: echo tmp102 0x48 > /sys/bus/i2c/devices/i2c-1/new_device

Q78: What is I2C bus arbitration?

Multi-master: if two masters start simultaneously, they monitor SDA. When a master sends 1 but sees 0, it lost arbitration and backs off. Bit-level arbitration — the master sending the lower address wins.

Q79: What is clock stretching in I2C?

Slave holds SCL low to pause master. Used when slave needs time to process data. Master must wait. Some masters don’t support it (e.g., bit-banged GPIO I2C on some SoCs).

Q80: How do you handle I2C devices without DT (e.g., for testing)?

Use sysfs: echo tmp102 0x48 > /sys/bus/i2c/devices/i2c-1/new_device creates i2c_client and triggers probe. Remove: echo 0x48 > /sys/bus/i2c/devices/i2c-1/delete_device.

Q81: What is the hwmon subsystem?

Standardized framework for hardware monitoring (temp, voltage, fan speed). Provides consistent sysfs interface (/sys/class/hwmon/). Integrates with sensors command. Temperature values in milli-degrees Celsius.

Q82: What is SPI chip select?

Dedicated line per SPI slave. Master asserts (low) CS before communication, deasserts after. Allows multiple devices on same bus. In DT: reg = <0> = chip select 0.

Q83: Explain SPI modes (CPOL, CPHA).

CPOL: clock polarity (idle high or low). CPHA: clock phase (sample on leading or trailing edge). Mode 0 (00): most common. Mode 3 (11): also common. Must match between master and slave.

Q84: What is I2C repeated start?

Send new START without STOP between transactions. Used for atomic read operations: Write register address → Repeated START → Read data. Prevents other masters from intervening.

Q85: How does Linux I2C device/driver matching work?

Kernel checks: 1) of_match_table (DT compatible string) — preferred. 2) id_table (static device name matching). 3) driver.name == device name. MODULE_DEVICE_TABLE(of, ...) creates alias for auto-loading.

CATEGORY 7: SYNCHRONIZATION & CONCURRENCY (20 Questions)

Q86: What types of locks exist in Linux kernel?

Spinlock: busy-wait, ISR-safe. Mutex: sleeping, process context only. Semaphore: counting, sleeping. RW spinlock: multiple readers OR one writer. RW semaphore: sleeping RW. Seqlock: optimistic writer, retry reader. RCU: lockless reads, deferred free.

Q87: When must you use spinlock vs mutex in kernel?

Spinlock: interrupt context, holding for < ~microsecond, can’t sleep. Mutex: process context, holding for longer, sleeping OK. NEVER: mutex in ISR (crash), spinlock + sleep (deadlock).

Q88: What is spin_lock_irqsave vs spin_lock?

spin_lock: disables preemption. spin_lock_irqsave: also saves and disables interrupts. Use irqsave when data is shared between process context and ISR. Without it: ISR could interrupt, try to lock same spinlock → deadlock.

Q89: What is RCU?

Read-Copy-Update. Readers: rcu_read_lock() (just disables preemption). Zero overhead reads. Writers: copy data, modify copy, atomically swap pointer (rcu_assign_pointer), wait for grace period (synchronize_rcu), free old copy. Used for: routing tables, process list, etc.

Q90: What is a completion in kernel?

Synchronization primitive for “wait until done” pattern. init_completion(&done). Waiter: wait_for_completion(&done). Signaler: complete(&done). One-shot synchronization. Used in: DMA completion, probe synchronization.

Q91: What is a seqlock?

Writer always proceeds (fast). Reader checks sequence counter before and after reading. If changed, retry. Good for: jiffies counter, time keeping. Writer can’t fail. Reader might retry.

Q92: What is wait_event / wake_up?

Kernel equivalent of condition variables. wait_event(wq, condition): sleeps until condition true. wake_up(&wq): wakes waiters to recheck condition. wait_event_interruptible: can be interrupted by signals.

Q93: What is a workqueue?

Defers work to kernel threads. INIT_WORK(&work, handler) + schedule_work(&work). Handler runs in process context (can sleep). Types: system_wq (shared), create_workqueue() (dedicated).

Q94: What is preempt_disable / preempt_enable?

Prevents current task from being preempted. Used to protect per-CPU data. Lighter than spinlock (no bus locking). Automatically done by spin_lock.

Q95: Explain the locking rules for data shared between process and interrupt context.

Process context: spin_lock_irqsave(&lock, flags). ISR: spin_lock(&lock). IRQsave in process context prevents local CPU from taking the interrupt while holding the lock (avoiding self-deadlock).

Q96: What is lockdep?

Kernel lock dependency validator (CONFIG_PROVE_LOCKING). Automatically detects potential deadlocks at runtime by tracking lock acquisition order. Essential development tool. Reports lock ordering violations even if deadlock hasn’t actually occurred yet.

Q97: What is a per-CPU variable?

Variable with separate copy for each CPU. No locking needed! DEFINE_PER_CPU(type, name). Access: this_cpu_read(name), this_cpu_write(name, val). Must disable preemption during access. Used for: statistics, counters, cached data.

CATEGORY 8: DEBUGGING & TOOLS (15 Questions)

Q98: How do you debug a kernel oops?

  1. Read the oops message (registers, stack trace). 2. Find the faulting instruction address. 3. Map to source: addr2line -e vmlinux ADDRESS. 4. Check for NULL pointer deref, buffer overflow, etc. 5. Enable CONFIG_DEBUG_INFO for better traces.

Q99: What is strace and when do you use it?

Traces system calls of a process. Shows each syscall, arguments, return value. Use for: understanding app-kernel interaction, debugging permission issues, finding missing files, profiling syscall overhead.

Q100: What is ftrace?

Kernel function tracer. Available at: /sys/kernel/debug/tracing/. Can trace every function call, measure latency, trace specific functions. echo function > current_tracer; cat trace.

Q101: How do you use gdb with kernel modules?

Debug loadable module: gdb vmlinux. add-symbol-file module.ko MODULE_LOAD_ADDRESS (from /sys/module/mymod/sections/.text). Set breakpoints. Requires KGDB or QEMU for actual debugging.

Q102: What is dmesg and its options?

Prints kernel ring buffer. dmesg -w: follow (like tail -f). dmesg -l err: filter by level. dmesg -T: human-readable timestamps. Messages from your pr_info() calls appear here.

Q103: What is perf?

Linux performance analysis tool. perf record ./app: capture profile. perf stat: count events. perf top: live function hotspots. Uses hardware performance counters and software tracepoints.

Q104: How do you detect memory errors in kernel?

KASAN (Kernel Address Sanitizer): detects out-of-bounds, use-after-free. KMEMLEAK: detects memory leaks. SLUB_DEBUG: slab corruption detection. All enabled via kernel config.

Q105: What tools help debug I2C issues?

i2cdetect -l: list buses. i2cdetect -y 1: scan bus 1. i2cdump -y 1 0x48: dump registers. i2cget/i2cset: read/write specific registers. dmesg for driver messages.

CATEGORY 9: REAL INTERVIEW SCENARIOS (15 Questions)

Q106: “Walk me through your character driver project.”

“I built a production-quality character device driver exposing /dev/chardev0. Architecture: user open() → VFS → my driver via file_operations. I use container_of() to get per-device state from inode’s cdev. Read/write uses copy_to/from_user for safe kernel↔user transfer. Concurrent access protected by mutex_lock_interruptible (so Ctrl+C works). I implemented IOCTL for control commands using _IO/_IOR/_IOW macros. Device nodes are auto-created via class_create + device_create + udev. I also exposed sysfs attributes for runtime monitoring.”

Q107: “Walk me through your I2C driver project.”

“I wrote a temperature sensor driver using the Linux I2C subsystem. Device Tree describes the sensor with compatible string ‘ti,tmp102’. When kernel matches this to my driver’s of_match_table, probe() is called. I use devm_kzalloc for managed allocation, configure the sensor via i2c_smbus_write_byte_data, and register with hwmon subsystem. Temperature reading uses i2c_smbus_read_word_swapped with 1-second caching to avoid bus flooding. I expose milli-celsius values through standard hwmon sysfs and custom attributes.”

Q108: “Your driver is crashing with a kernel oops on read(). How do you debug it?”

  1. Check dmesg for oops trace — find faulting instruction. 2. Use addr2line to map to source. 3. Common causes: NULL private_data (forgot to set in open?), buffer overrun, wrong pointer arithmetic. 4. Add pr_info() at entry/exit of read(). 5. Check if f_pos > buffer size. 6. Verify copy_to_user arguments. 7. Enable CONFIG_KASAN for memory errors.

Q109: “How would you handle a device that’s shared between multiple processes?”

“Each open() gets its own struct file with separate f_pos. They all share the device buffer via private_data. I protect concurrent access with mutex. For exclusive access: check a flag in open(), return -EBUSY if already opened. For truly independent access: allocate per-open-instance buffers in open(), store in private_data.”

Q110: “Design a driver for a device that needs fast interrupt response.”

“Use threaded IRQ: fast top-half acknowledges interrupt and reads critical data, threaded bottom-half does processing. Top half: spin_lock_irqsave for shared data, return IRQ_WAKE_THREAD. Thread handler: can use mutex, I2C/SPI transactions, workqueues. Use DMA for bulk data transfer to avoid PIO in ISR.”

Q111: “How do you ensure your driver doesn’t leak memory?”

“1. Use devm_* functions for all allocations — auto-freed on error/remove. 2. Follow goto cleanup pattern: allocate in order, free in reverse on error. 3. Test with KMEMLEAK enabled. 4. Code review: every alloc has matching free. 5. Error path testing: deliberately fail each error path.”

Q112: “You need to add support for a new revision of your sensor chip. How?”

“1. Add new compatible string to of_match_table with config data pointer for new revision. 2. In probe(), of_device_get_match_data() returns version-specific config. 3. Use config struct for differences (register offsets, conversion factors, features). 4. No if/else chains — data-driven approach scales cleanly.”

Q113: “Your driver works in test but fails on different hardware. What do you check?”

“1. DT: correct register addresses, interrupt numbers? 2. Clock/power: device clocked and powered? Check regulator/clock DT bindings. 3. Pin mux: correct pin function configured? 4. I2C address: check with i2cdetect. 5. Pull-ups: I2C needs them. 6. Timing: check bus speed configuration.”

Q114: “Explain the difference between platform_driver and i2c_driver.”

“Both use probe/remove model. platform_driver: for SoC-internal peripherals addressable via MMIO (timers, GPIO, UART). Resources via platform_get_resource. i2c_driver: for I2C bus devices. Receives i2c_client with bus address. Uses i2c_smbus_* for communication. Also: spi_driver for SPI devices.”

Q115: “How do you make your driver debuggable in production?”

“1. Use dev_dbg() — disabled by default, enabled with dyndbg. 2. Sysfs attributes for status/stats monitoring. 3. debugfs files for internal state. 4. Tracepoints for ftrace integration. 5. Module parameter for debug level: module_param(debug, int, 0644). 6. Meaningful error messages with dev_err().”

CATEGORY 10: SYSTEM DESIGN & ARCHITECTURE (10 Questions)

Q116: How does Linux support thousands of different hardware platforms?

“Uses a bus/device/driver abstraction. Drivers register with bus subsystem (platform, I2C, SPI, PCI, USB). Hardware described by Device Tree (ARM) or ACPI (x86). Kernel matches devices to drivers at runtime. Same driver binary works across platforms — only DT/ACPI changes.”

Q117: What is the Linux Driver Model (LDM)?

“Unified framework: struct bus_type, struct device, struct device_driver. Bus matches device to driver. probe() called on match. Sysfs automatically exposes hierarchy. Power management hooks. Consistent across all bus types.”

Q118: How do you choose between polling and interrupt-driven I/O?

“Interrupts: for infrequent or unpredictable events (button press, packet arrival). Efficient — CPU does other work. Polling: for high-frequency data (high-speed ADC), when latency must be deterministic, or in ISR when can’t sleep. Hybrid: NAPI (network) — interrupt triggers poll mode during bursts.”

Q119: Design a driver for a custom SoC peripheral.

“1. Study hardware spec (registers, interrupts, DMA). 2. Write DT node with base address, IRQ, clocks. 3. Create platform_driver with of_match_table. 4. In probe: devm_platform_ioremap_resource, devm_request_irq, register subsystem (hwmon/input/net/etc.). 5. Implement file_ops or subsystem callbacks. 6. Test on real hardware with dmesg + ftrace.”

Q120: What is Bus Mastering and why do we need DMA?

“CPU-driven I/O (PIO): CPU reads/writes each byte. Blocks CPU. DMA: hardware DMA controller reads/writes memory independently. CPU does other work. When done, DMA raises interrupt. Critical for: network, storage, multimedia. Modern SoCs have multiple DMA channels.”

fail_res1: free1(res1);
fail:      return ret;
}
