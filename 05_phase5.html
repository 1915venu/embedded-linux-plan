PHASE 5: DEVICE TREE & PLATFORM DRIVERS (Days 23-27)

DAYS 23-24: DEVICE TREE FUNDAMENTALS

🧠 THEORY: Why Device Tree Exists

BEFORE Device Tree (pre-2012):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
arm/mach-omap2/board-beagle.c     ← Board-specific C code
arm/mach-omap2/board-panda.c      ← Another board
arm/mach-exynos/mach-smdk.c       ← Samsung board
arm/mach-tegra/board-harmony.c    ← NVIDIA board
...
HUNDREDS of board files, each compiling different kernel!

Linus Torvalds (2011): "This is a f***ing mess"

AFTER Device Tree:
━━━━━━━━━━━━━━━━━━
One kernel binary (vmlinuz) works on ALL boards.
Hardware differences described in .dtb (Device Tree Blob):
  arch/arm/boot/dts/am335x-boneblack.dts   ← BeagleBone Black
  arch/arm/boot/dts/sun8i-h3-orangepi.dts  ← Orange Pi
  arch/arm/boot/dts/bcm2711-rpi-4-b.dts    ← Raspberry Pi 4

Bootloader loads correct .dtb for the board → Kernel configures itself

🧠 THEORY: Device Tree Syntax

/* A DTS (Device Tree Source) file is a tree of nodes */
/dts-v1/;

/ {    /* Root node */
    model = "My Custom Board";
    compatible = "myvendor,myboard";

    /* Each node represents a hardware component */
    /* Node name format: name@unit-address */

    memory@80000000 {
        device_type = "memory";
        reg = <0x80000000 0x20000000>;  /* base=0x8000_0000, size=512MB */
    };

    /* Peripherals section */
    soc {
        compatible = "simple-bus";
        #address-cells = <1>;   /* Child reg addresses are 1 cell (32-bit) */
        #size-cells = <1>;      /* Child reg sizes are 1 cell (32-bit) */
        ranges;                 /* 1:1 address translation */

        uart0: serial@44e09000 {
            compatible = "ti,am335x-uart";   /* KEY: matches driver! */
            reg = <0x44e09000 0x2000>;       /* Base address, size */
            interrupts = <72>;               /* IRQ number */
            clock-frequency = <48000000>;    /* 48 MHz */
            status = "okay";                 /* enabled */
        };

        i2c0: i2c@44e0b000 {
            compatible = "ti,omap4-i2c";
            reg = <0x44e0b000 0x1000>;
            interrupts = <70>;
            #address-cells = <1>;           /* I2C child = address only */
            #size-cells = <0>;              /* No size for I2C */
            clock-frequency = <400000>;     /* 400 kHz fast mode */
            status = "okay";

            /* Devices ON the I2C bus (children) */
            temp_sensor: tmp102@48 {
                compatible = "ti,tmp102";   /* Matches I2C driver! */
                reg = <0x48>;               /* I2C address (7-bit) */
                #io-channel-cells = <0>;
            };

            eeprom@50 {
                compatible = "atmel,24c256";
                reg = <0x50>;
            };
        };

        gpio0: gpio@44e07000 {
            compatible = "ti,omap4-gpio";
            reg = <0x44e07000 0x1000>;
            interrupts = <96>;
            gpio-controller;
            #gpio-cells = <2>;              /* gpio num + flags */
        };
    };

    /* LEDs described in DT */
    leds {
        compatible = "gpio-leds";

        heartbeat {
            label = "my-led";
            gpios = <&gpio0 14 GPIO_ACTIVE_HIGH>;
            linux,default-trigger = "heartbeat";
        };
    };
};

🧠 THEORY: Key Properties Reference

Property Purpose Example
compatible Matches device to driver "ti,tmp102"
reg Register addresses & sizes / bus address <0x44e09000 0x2000>
interrupts Interrupt specification <GIC_SPI 45 IRQ_TYPE_LEVEL_HIGH>
status Enable/disable device "okay" or "disabled"
clocks Clock references <&clock_ctrl 5>
clock-frequency Clock speed <400000>
#address-cells Address width for children <1> = 32-bit
#size-cells Size width for children <0> = no size
ranges Address translation Empty = 1:1
phandle Reference to another node <&gpio0>

🧠 THEORY: Compatible String Matching — The Heart of DT

How matching works:

Device Tree node:
    temp_sensor {
        compatible = "ti,tmp102", "ti,tmp100", "generic-temp";
    };                  ↑ most specific       ↑ fallback

Driver code:
    static const struct of_device_id my_match[] = {
        { .compatible = "ti,tmp102", .data = &tmp102_config },
        { .compatible = "generic-temp", .data = &generic_config },
        { }  ← terminator
    };

Matching process (kernel tries in order):
1. Try "ti,tmp102" → FOUND in driver! Use tmp102_config ✓
2. (If not found, try "ti,tmp100") → Not in driver
3. (If not found, try "generic-temp") → Found in driver as fallback

BEST PRACTICE for compatible:
"vendor,specific-version", "vendor,generic-version"

💻 COMPLETE CODE: DT Property Reading Demo

// File: ~/embedded_learning/day_23/dt_demo/dt_demo.c
// This platform driver demonstrates reading DT properties
// It won't match real hardware — it's for learning the APIs

#include <linux/init.h>
#include <linux/module.h>
#include <linux/platform_device.h>
#include <linux/of.h>
#include <linux/of_device.h>

MODULE_LICENSE("GPL");

static int dt_demo_probe(struct platform_device *pdev)
{
    struct device *dev = &pdev->dev;
    struct device_node *np = dev->of_node;
    u32 value;
    const char *str;
    int ret;

    dev_info(dev, "Probe called!\n");

    /* Read single u32 property */
    ret = of_property_read_u32(np, "my-frequency", &value);
    if (ret == 0)
        dev_info(dev, "Frequency: %u Hz\n", value);
    else
        dev_info(dev, "No frequency property\n");

    /* Read string property */
    ret = of_property_read_string(np, "my-label", &str);
    if (ret == 0)
        dev_info(dev, "Label: %s\n", str);

    /* Read boolean property (presence = true) */
    if (of_property_read_bool(np, "my-feature-enabled"))
        dev_info(dev, "Feature is ENABLED\n");
    else
        dev_info(dev, "Feature is DISABLED\n");

    /* Read array */
    u32 arr[4];
    int count = of_property_count_u32_elems(np, "my-array");
    if (count > 0 && count <= 4) {
        of_property_read_u32_array(np, "my-array", arr, count);
        dev_info(dev, "Array: %u %u %u %u (%d elements)\n",
                 arr[0], count > 1 ? arr[1] : 0,
                 count > 2 ? arr[2] : 0,
                 count > 3 ? arr[3] : 0, count);
    }

    /* Get version-specific data from compatible match */
    const struct of_device_id *match = of_match_device(
        pdev->dev.driver->of_match_table, dev);
    if (match && match->data)
        dev_info(dev, "Match data pointer: %p\n", match->data);

    dev_info(dev, "Probe complete!\n");
    return 0;
}

static int dt_demo_remove(struct platform_device *pdev)
{
    dev_info(&pdev->dev, "Remove called\n");
    return 0;
}

static const struct of_device_id dt_demo_of_match[] = {
    { .compatible = "demo,dt-learning", },
    { }
};
MODULE_DEVICE_TABLE(of, dt_demo_of_match);

static struct platform_driver dt_demo_driver = {
    .probe = dt_demo_probe,
    .remove = dt_demo_remove,
    .driver = {
        .name = "dt_demo",
        .of_match_table = dt_demo_of_match,
    },
};

module_platform_driver(dt_demo_driver);

DAYS 25-27: PLATFORM DRIVER

🧠 THEORY: Platform Driver Model

In the kernel, there are multiple BUS TYPES:

┌──────────────────────────────────────────────────────┐
│                  Linux Driver Model                   │
│                                                       │
│  ┌─────────┐  ┌──────┐  ┌──────┐  ┌─────┐  ┌─────┐ │
│  │Platform  │  │ I2C  │  │ SPI  │  │ PCI │  │ USB │ │
│  │  Bus     │  │ Bus  │  │ Bus  │  │ Bus │  │ Bus │ │
│  └────┬────┘  └───┬──┘  └───┬──┘  └──┬──┘  └──┬──┘ │
│       │           │         │        │        │     │
│  On-SoC       I2C devices  SPI     PCIe    USB     │
│  peripherals  (sensors,   devices  cards   devices  │
│  (UART, GPIO, EEPROM)     (flash)                   │
│  Timer, DMA)                                         │
└──────────────────────────────────────────────────────┘

Platform Bus = for devices directly connected to SoC
  - Addressable via memory-mapped I/O
  - Not on a discoverable bus (unlike PCI/USB)
  - Described in Device Tree or ACPI

🧠 THEORY: probe() and remove() — The Full Story

DRIVER REGISTRATION                    DEVICE DISCOVERY
(when module loads)                    (from Device Tree)
     │                                      │
     │  module_platform_driver()            │  Kernel parses DTB
     │  ├── registers with platform bus     │  ├── Creates platform_device
     │  ├── provides of_match_table         │  │   for each DT node
     │  └── provides probe/remove           │  └── Adds to platform bus
     │                                      │
     └──────────────────┬───────────────────┘
                        │
                        ▼
              MATCHING (by platform bus)
              Compare device's compatible string
              with driver's of_match_table
                        │
                        ▼ (MATCH!)
              probe(platform_device *pdev)
              ├── Get resources from DT:
              │   ├── devm_platform_ioremap_resource()  → registers
              │   ├── platform_get_irq()                → IRQ
              │   ├── devm_clk_get()                    → clock
              │   └── of_property_read_*()              → custom props
              ├── Initialize hardware
              ├── Register with subsystem
              │   └── (hwmon, input, net, char device, etc.)
              └── Return 0 (success) or negative error

              remove(platform_device *pdev)
              ├── Unregister from subsystem
              ├── Release resources
              │   (devm_* resources auto-freed!)
              └── Return 0

// File: ~/embedded_learning/day_25/platform_drv/myplat.c
#include <linux/init.h>
#include <linux/module.h>
#include <linux/platform_device.h>
#include <linux/of.h>
#include <linux/of_device.h>
#include <linux/io.h>

MODULE_LICENSE("GPL");
MODULE_DESCRIPTION("Platform driver learning example");

/* Version-specific configuration via match data */
struct mydev_config {
    const char *name;
    int features;
    int max_speed;
};

static const struct mydev_config v1_config = {
    .name = "MyDevice-v1",
    .features = 3,
    .max_speed = 100000,
};

static const struct mydev_config v2_config = {
    .name = "MyDevice-v2",
    .features = 7,
    .max_speed = 400000,
};

/* Per-device instance data */
struct mydev_data {
    void __iomem *base;      /* Memory-mapped register base */
    int irq;                 /* IRQ number */
    const struct mydev_config *config;
    struct device *dev;
};

static int mydev_probe(struct platform_device *pdev)
{
    struct device *dev = &pdev->dev;
    struct mydev_data *data;
    struct resource *res;
    int ret;

    dev_info(dev, "Probe starting\n");

    /* Allocate per-device data (managed — auto-freed!) */
    data = devm_kzalloc(dev, sizeof(*data), GFP_KERNEL);
    if (!data)
        return -ENOMEM;

    data->dev = dev;

    /* Get version-specific config from DT match */
    data->config = of_device_get_match_data(dev);
    if (!data->config) {
        dev_err(dev, "No match data!\n");
        return -ENODEV;
    }
    dev_info(dev, "Matched: %s (features=%d, speed=%d)\n",
             data->config->name, data->config->features,
             data->config->max_speed);

    /* Get memory resource from DT 'reg' property */
    /* devm_platform_ioremap_resource does:
     * 1. platform_get_resource(pdev, IORESOURCE_MEM, 0)
     * 2. devm_request_mem_region()
     * 3. devm_ioremap()
     * All managed — auto-cleaned-up!
     */
    data->base = devm_platform_ioremap_resource(pdev, 0);
    if (IS_ERR(data->base)) {
        /* On simulated/VM environment, this is expected to fail */
        dev_warn(dev, "No memory resource (expected in VM)\n");
        data->base = NULL;
    } else {
        dev_info(dev, "Mapped registers at %p\n", data->base);
        /* Access registers: readl(data->base + OFFSET) */
    }

    /* Get IRQ from DT 'interrupts' property */
    data->irq = platform_get_irq(pdev, 0);
    if (data->irq > 0) {
        dev_info(dev, "IRQ: %d\n", data->irq);
        /* Would devm_request_irq here */
    } else {
        dev_warn(dev, "No IRQ (expected in VM)\n");
    }

    /* Read custom DT properties */
    u32 custom_val;
    ret = of_property_read_u32(dev->of_node, "custom-property", &custom_val);
    if (ret == 0)
        dev_info(dev, "Custom property: %u\n", custom_val);

    /* Store driver data in platform device */
    platform_set_drvdata(pdev, data);

    dev_info(dev, "Probe complete!\n");
    return 0;
}

static int mydev_remove(struct platform_device *pdev)
{
    struct mydev_data *data = platform_get_drvdata(pdev);

    /* With devm_* functions, most cleanup is automatic!
     * Only need to unregister from subsystems here */

    dev_info(data->dev, "Removed (%s)\n", data->config->name);
    return 0;
}

/* DT compatible matching table */
static const struct of_device_id mydev_of_match[] = {
    { .compatible = "myvendor,mydevice-v1", .data = &v1_config },
    { .compatible = "myvendor,mydevice-v2", .data = &v2_config },
    { }
};
MODULE_DEVICE_TABLE(of, mydev_of_match);

/* Platform driver registration */
static struct platform_driver mydev_driver = {
    .probe = mydev_probe,
    .remove = mydev_remove,
    .driver = {
        .name = "mydevice",
        .of_match_table = mydev_of_match,
    },
};

/* This macro creates init/exit that register/unregister the driver */
module_platform_driver(mydev_driver);

📝 INTERVIEW QUESTIONS — Device Tree & Platform (16 Questions)

Q1: What is Device Tree? Why was it introduced?

Hardware description separate from code. Before DT: every ARM board = custom C file = separate kernel compile. DT lets one kernel binary work on many boards. Only the .dtb file changes per hardware.

Q2: What is the compatible property and how does matching work?

String list identifying device. Driver provides of_match_table. Kernel tries each compatible string until match found. First = most specific. Allows fallback matching.

Q3: What is #address-cells and #size-cells?

Specify format of reg property in child nodes. address-cells=1: 32-bit addresses. size-cells=0: no size (I2C addresses are just addresses). size-cells=1: 32-bit sizes.

Q4: What is devm_platform_ioremap_resource?

One function that: gets resource from DT, requests memory region (prevents conflicts), maps physical to kernel virtual address. All managed — freed on error/remove. Replaces 3 separate calls.

Q5: What is platform_get_irq?

Gets IRQ number from DT interrupts property. Handles IRQ domain translation. Returns IRQ number ready for request_irq().

Q6: Explain of_device_get_match_data.

Returns .data pointer from matching of_device_id entry. Use for version-specific config without runtime string comparison. Kernel already did matching — just retrieve the data.

Q7: What is module_platform_driver macro?

Creates __init and __exit functions that register/unregister the platform driver. Equivalent to: init → platform_driver_register(), exit → platform_driver_unregister(). Less boilerplate.

Q8: Platform driver vs character driver — relationship?

Platform driver handles device discovery (probe/remove via DT). Character driver handles user-space interface (read/write/ioctl). A real driver often IS BOTH: platform driver for HW interaction, creates chardev for user interface.

Q9: What is deferred probing and when does it happen?

probe() returns -EPROBE_DEFER when a dependency isn’t ready (clock, regulator, GPIO). Kernel queues the device and retries later. Essential for non-deterministic driver initialization order.

Q10: How do you handle device tree overlays?

Overlays modify base DT at runtime. Used for: hardware add-ons (RPi HATs), enabling peripherals, hot-pluggable devices. Applied via configfs: /sys/kernel/config/device-tree/overlays/.

Q11: What is the ranges property?

Address translation between parent and child. Empty ranges = 1:1 mapping (child addresses = parent addresses). Non-empty: specifies translation offsets.

Q12: How does your driver know the register base address?

From DT reg property. In probe: devm_platform_ioremap_resource(pdev, 0) reads first reg entry. Returns virtual address. Access registers with readl/writel.

Q13: What is MMIO? How do you access it safely?

Memory-Mapped I/O: hardware registers at specific physical addresses. In driver: ioremap maps to kernel virtual address. Use readl()/writel() (not pointer dereference!) — they include memory barriers.

Q14: Difference between readl/writel and ioread32/iowrite32?

readl/writel: for MMIO (memory-mapped). ioread32/iowrite32: work for both MMIO and port I/O (x86). Modern drivers should use ioread/iowrite for portability.

Q15: What happens if your driver’s probe fails?

Return negative errno. devm_* resources auto-freed. Device stays unbound. dmesg shows error. User sees device in /sys/bus/platform/devices/ but not bound to driver.

Q16: How do you test a platform driver without real hardware?

  1. DT overlay to create platform device. 2. QEMU with custom DTB. 3. Use sysfs to manually bind: echo "mydevice" > /sys/bus/platform/drivers/.../bind. 4. Module parameter to skip HW checks in probe.
