PHASE 2: LINUX FUNDAMENTALS (Days 4-8)

DAY 4: SYSTEM CALLS — THE KERNEL BOUNDARY

🧠 THEORY: User Space vs Kernel Space (Why It Exists)

The entire Linux architecture is built on this fundamental split:

User Space (Ring 3)                    Kernel Space (Ring 0)
┌──────────────────────────┐          ┌──────────────────────────────────┐
│  Your application        │          │  Full hardware access            │
│  Limited privileges      │          │  All memory accessible           │
│  Virtual addresses only  │   ───►   │  Physical + Virtual memory       │
│  Can crash safely        │  syscall │  Bug here = system crash         │
│  Multiple processes      │          │  Single address space (shared)   │
│  ~3GB address space (32) │          │  ~1GB address space (32-bit)     │
└──────────────────────────┘          └──────────────────────────────────┘

Why the split?

  1. Protection: User code can’t corrupt kernel or other processes
  2. Stability: App crash doesn’t take down the system
  3. Security: Prevents malicious code from accessing hardware directly
  4. Abstraction: Same syscall interface regardless of hardware

🧠 THEORY: System Call Mechanism — Step by Step

Application: write(fd, buf, count)
    │
    │ 1. glibc wrapper loads syscall number into register
    │    (write = syscall #1 on x86_64)
    │
    │ 2. Arguments loaded into registers:
    │    rdi = fd, rsi = buf, rdx = count
    │
    │ 3. SYSCALL instruction (x86_64) or int 0x80 (x86_32)
    │    ┌─────────────────────────────────────────┐
    │    │ CPU switches to Ring 0                   │
    │    │ Stack switches to kernel stack            │
    │    │ Saves user registers on kernel stack      │
    │    └─────────────────────────────────────────┘
    │
    │ 4. Kernel entry point (entry_SYSCALL_64 in entry/entry_64.S)
    │
    │ 5. Look up handler: sys_call_table[__NR_write] → ksys_write()
    │
    │ 6. Handler validates parameters
    │    - Is fd valid?
    │    - Is buf a valid user pointer?
    │    - Does process have permission?
    │
    │ 7. Perform the operation
    │
    │ 8. Return value placed in rax
    │    If negative = errno value
    │
    │ 9. SYSRET instruction
    │    ┌─────────────────────────────────────────┐
    │    │ CPU switches back to Ring 3               │
    │    │ Stack switches back to user stack          │
    │    │ Restores user registers                    │
    │    └─────────────────────────────────────────┘
    │
    ▼ Application continues with return value

🧠 THEORY: Key System Calls for Embedded

System Call Purpose Kernel Function
open() Open file/device ksys_open() → VFS → driver’s .open()
read() Read data → VFS → driver’s .read()
write() Write data → VFS → driver’s .write()
close() Close FD → VFS → driver’s .release()
ioctl() Device control → VFS → driver’s .unlocked_ioctl()
mmap() Map memory → VFS → driver’s .mmap()
poll() Wait for events → VFS → driver’s .poll()
fork() Create process → kernel_clone()
execve() Replace process → Load new ELF binary

Key insight: When user calls open("/dev/mydevice"), it eventually calls YOUR driver’s .open() function!

🧠 THEORY: File Descriptors — The Universal Handle

Process File Descriptor Table:
┌────┬──────────────────────────────┐
│ FD │ Points to...                  │
├────┼──────────────────────────────┤
│  0 │ → Open File Entry (stdin)    │ → /dev/tty
│  1 │ → Open File Entry (stdout)   │ → /dev/tty
│  2 │ → Open File Entry (stderr)   │ → /dev/tty
│  3 │ → Open File Entry            │ → /dev/mydevice
│  4 │ → Open File Entry            │ → /home/user/data.txt
│  5 │ → (unused)                   │
└────┴──────────────────────────────┘

Each Open File Entry contains:
- Current position (offset)
- Open flags (O_RDONLY, etc.)
- Pointer to inode (which file)
- Pointer to file_operations (which driver)

#define CHECK(x, msg) do { \
    if ((x) < 0) { \
        fprintf(stderr, "[ERROR] %s: %s (errno=%d)\n", msg, strerror(errno), errno); \
        exit(EXIT_FAILURE); \
    } \
} while(0)

// Robust copy with full error handling
int copy_file(const char *src, const char *dst) {
    int src_fd, dst_fd;
    char buffer[4096];
    ssize_t bytes_read, bytes_written, total_written;

    src_fd = open(src, O_RDONLY);
    if (src_fd < 0) return -1;

    dst_fd = open(dst, O_WRONLY | O_CREAT | O_TRUNC, 0644);
    if (dst_fd < 0) {
        close(src_fd);
        return -1;
    }

    while ((bytes_read = read(src_fd, buffer, sizeof(buffer))) > 0) {
        total_written = 0;
        while (total_written < bytes_read) {
            // Handle partial writes (important for devices/pipes!)
            bytes_written = write(dst_fd, buffer + total_written,
                                 bytes_read - total_written);
            if (bytes_written < 0) {
                if (errno == EINTR) continue;  // Interrupted by signal, retry
                close(src_fd);
                close(dst_fd);
                return -1;
            }
            total_written += bytes_written;
        }
    }

    close(src_fd);
    close(dst_fd);
    return (bytes_read < 0) ? -1 : 0;
}

// Demonstrate lseek
void demo_lseek(const char *path) {
    int fd = open(path, O_RDONLY);
    if (fd < 0) { perror("open"); return; }

    // Get file size using lseek
    off_t size = lseek(fd, 0, SEEK_END);
    printf("File size via lseek: %ld bytes\n", size);

    // Read last 10 bytes
    lseek(fd, -10, SEEK_END);
    char buf[11] = {0};
    read(fd, buf, 10);
    printf("Last 10 bytes: '%s'\n", buf);

    close(fd);
}

// File info via stat
void print_file_info(const char *path) {
    struct stat st;
    if (stat(path, &st) < 0) { perror("stat"); return; }

    printf("File: %s\n", path);
    printf("  Size: %ld bytes\n", st.st_size);
    printf("  Mode: %o\n", st.st_mode & 0777);
    printf("  Type: ");
    if (S_ISREG(st.st_mode))  printf("Regular file\n");
    else if (S_ISDIR(st.st_mode))  printf("Directory\n");
    else if (S_ISCHR(st.st_mode))  printf("Character device\n");
    else if (S_ISBLK(st.st_mode))  printf("Block device\n");
    else if (S_ISFIFO(st.st_mode)) printf("FIFO\n");
    else printf("Other\n");
    printf("  Inode: %lu\n", st.st_ino);
    printf("  Links: %lu\n", (unsigned long)st.st_nlink);
}

📝 INTERVIEW QUESTIONS — Day 4 (10 Questions)

Q1: What is the difference between open() and fopen()?

open(): System call, returns int fd, no buffering, for device files & precise control. fopen(): C library function, returns FILE*, provides buffering, for regular file I/O.

Q2: Explain the file descriptor table.

Each process has FD table mapping integers to kernel file objects. fd 0=stdin, 1=stdout, 2=stderr. open() returns lowest available fd. FDs are per-process, but the underlying file objects can be shared (after fork() or dup()).

Q3: What happens when you call write() on a device file?

write() → syscall → VFS layer → looks up device by major number → calls driver’s .write() function via file_operations. Your driver then handles the data.

Q4: What is strace and how does it work?

strace uses ptrace() syscall to intercept all system calls made by a process. Shows syscall name, arguments, and return value. Essential debugging tool for understanding app-kernel interaction.

Q5: Explain dup() and dup2().

dup(fd) creates a copy of fd (new fd pointing to same file). dup2(oldfd, newfd) makes newfd a copy of oldfd, closing newfd first. Used for I/O redirection (e.g., redirecting stdout to a file).

Q6: What is the difference between stat(), fstat(), and lstat()?

stat(path): gets info by path. fstat(fd): gets info by fd (already opened). lstat(path): like stat but doesn’t follow symlinks. In kernel driver context, you usually work with fd (inode already known).

Q7: What flags would you use to open a file for appending?

O_WRONLY | O_APPEND | O_CREAT with mode 0644. O_APPEND seeks to end before each write atomically. Multiple processes can append safely.

Q8: What is the significance of O_NONBLOCK?

Makes file operations non-blocking. If data isn’t available, read() returns EAGAIN instead of blocking. Critical for device drivers and event-driven programming. Kernel driver checks filp->f_flags & O_NONBLOCK.

Q9: How does the kernel know which driver handles a device file?

Device files have major and minor numbers (visible with ls -l /dev/). Major number identifies the driver. Minor number identifies the specific device instance. The kernel maintains a table mapping major numbers to registered drivers.

Q10: What is the VFS layer?

Virtual File System Switch — abstraction layer that provides uniform file interface regardless of underlying filesystem or device. All file operations go through VFS, which then dispatches to the appropriate filesystem driver or device driver based on the file type.

DAY 5: PROCESS CREATION (FORK & EXEC)

🧠 THEORY: Process Creation in Linux

Linux creates processes using a unique 2-step model:

Step 1: fork() — Clone the current process - Creates an almost-exact copy (child) - Uses Copy-On-Write (COW): pages shared until written - Returns twice: 0 to child, child PID to parent

Step 2: exec() — Replace the process image - Replaces code, data, stack with new program - Same PID, same file descriptors (unless FD_CLOEXEC set) - This is why shell commands work: shell forks, child execs

🧠 THEORY: Copy-On-Write (COW) — Performance Critical

Before fork():
Parent virtual page → Physical page P1 (writable)

After fork():
Parent virtual page → Physical page P1 (read-only)
Child  virtual page → Physical page P1 (read-only)  ← SAME physical page!

When child writes:
Page fault! → Kernel copies P1 → P2
Parent virtual page → P1 (writable again)
Child  virtual page → P2 (writable, modified copy)

Why COW matters: 1. Fork is fast — just copy page table entries, not pages 2. Child often calls exec() immediately — no need to copy at all 3. Read-only data (code) is never copied

// File: ~/embedded_learning/day_05/process_complete.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>
#include <sys/types.h>
#include <string.h>
#include <errno.h>

// Pattern 1: Simple fork
void demo_fork() {
    printf("\n=== SIMPLE FORK ===\n");
    int shared_var = 42;

    pid_t pid = fork();

    if (pid < 0) {
        perror("fork failed");
        exit(1);
    } else if (pid == 0) {
        // Child modifies shared_var — only its copy (COW)
        shared_var = 99;
        printf("CHILD: PID=%d, PPID=%d, shared_var=%d (our copy)\n",
               getpid(), getppid(), shared_var);
        exit(0);
    } else {
        wait(NULL);
        printf("PARENT: PID=%d, Child=%d, shared_var=%d (unchanged!)\n",
               getpid(), pid, shared_var);
    }
}

// Pattern 2: Fork + Exec
void demo_fork_exec() {
    printf("\n=== FORK + EXEC ===\n");

    pid_t pid = fork();

    if (pid == 0) {
        printf("Child about to exec 'ls -la'...\n");
        execl("/bin/ls", "ls", "-la", "/tmp", NULL);
        // Only reached if exec fails!
        perror("exec failed");
        exit(1);
    } else if (pid > 0) {
        int status;
        waitpid(pid, &status, 0);

        if (WIFEXITED(status))
            printf("Child exited normally, status=%d\n", WEXITSTATUS(status));
        else if (WIFSIGNALED(status))
            printf("Child killed by signal %d\n", WTERMSIG(status));
    }
}

// Pattern 3: Pipe communication
void demo_pipe() {
    printf("\n=== PIPE COMMUNICATION ===\n");

    int pipefd[2];  // pipefd[0]=read, pipefd[1]=write

    if (pipe(pipefd) < 0) { perror("pipe"); return; }

    pid_t pid = fork();

    if (pid == 0) {
        close(pipefd[1]);  // Close write end in child
        char buffer[256];
        ssize_t n = read(pipefd[0], buffer, sizeof(buffer)-1);
        buffer[n] = '\0';
        printf("Child received: '%s'\n", buffer);
        close(pipefd[0]);
        exit(0);
    } else {
        close(pipefd[0]);  // Close read end in parent
        const char *msg = "Hello from parent via pipe!";
        write(pipefd[1], msg, strlen(msg));
        close(pipefd[1]);  // Signal EOF to child
        wait(NULL);
    }
}

// Pattern 4: Multiple children
void demo_multi_child() {
    printf("\n=== MULTIPLE CHILDREN ===\n");
    const int NUM_CHILDREN = 3;
    pid_t children[NUM_CHILDREN];

    for (int i = 0; i < NUM_CHILDREN; i++) {
        children[i] = fork();
        if (children[i] == 0) {
            printf("Child %d (PID=%d) starting work...\n", i, getpid());
            usleep((3 - i) * 100000);  // Different durations
            printf("Child %d done\n", i);
            exit(i);  // Exit with different codes
        }
    }

    // Parent waits for all children
    for (int i = 0; i < NUM_CHILDREN; i++) {
        int status;
        pid_t pid = waitpid(children[i], &status, 0);
        printf("Child PID=%d exited with status=%d\n", pid, WEXITSTATUS(status));
    }
}

int main() {
    demo_fork();
    demo_fork_exec();
    demo_pipe();
    demo_multi_child();
    return 0;
}

📝 INTERVIEW QUESTIONS — Day 5 (12 Questions)

Q1: What does fork() return?

Returns 0 to child, child’s PID to parent, -1 on error. The process execution continues from the same point in both parent and child.

Q2: What is a zombie process?

A finished process whose exit status hasn’t been read by parent via wait(). Takes no memory/CPU but occupies a PID slot. Parent MUST call wait() or set SIGCHLD to SIG_IGN.

Q3: What is an orphan process?

A process whose parent terminated. Adopted by init (PID 1) or subreaper. Init calls wait() when orphan exits, preventing zombies.

Q4: Explain Copy-On-Write.

After fork(), parent and child share same physical pages (marked read-only). On first write, page fault triggers kernel to copy the page. Huge optimization — if child calls exec() immediately, no copying needed at all.

Q5: Difference between exec family functions?

execl: list of args. execv: vector (array) of args. execlp/execvp: search PATH. execle/execve: pass environment. All replace current process image.

Q6: What happens to open file descriptors after fork()?

Child inherits copies of all parent’s file descriptors. Both share the same file position (offset). Writing from either changes the shared offset. This is why pipes work — parent and child share pipe fds.

Q7: What is wait() vs waitpid()?

wait(): waits for any child, blocks. waitpid(pid, &status, flags): waits for specific child, can be non-blocking with WNOHANG. Can check specific child or process group.

Q8: How does a shell implement piping (ls | grep foo)?

Shell creates pipe, forks twice. First child: dup2(pipe_write, STDOUT), exec(“ls”). Second child: dup2(pipe_read, STDIN), exec(“grep”). Parent closes pipe, waits for both.

Q9: What is the difference between _exit() and exit()?

exit(): flushes stdio buffers, calls atexit() handlers, then calls _exit(). _exit(): immediate termination, no cleanup. After fork(), child should use _exit() to avoid flushing parent’s buffers.

Q10: What is vfork()?

Like fork() but child shares parent’s address space (no COW). Parent is suspended until child calls exec() or _exit(). Faster than fork() but dangerous — child must not modify shared memory. Largely obsolete with efficient fork() + COW.

Q11: How do kernel threads differ from user processes?

Kernel threads (kthread_create()) run only in kernel space, no user-space address space. Share kernel address space. Used for background tasks: kworker, ksoftirqd, kswapd.

Q12: What is the clone() system call?

Underlying syscall for both fork() and pthread_create(). Flags control what is shared: CLONE_VM (share memory), CLONE_FILES (share fd table), etc. Most flexible Linux process/thread creation mechanism.

DAY 6: MEMORY MAPPED FILES (MMAP)

🧠 THEORY: Why mmap() Matters

mmap() is one of the most important system calls for embedded Linux because: 1. Device drivers use it to map hardware registers into user space 2. Shared memory between processes uses it 3. Large file access without read/write overhead 4. DMA buffers are exposed to user space via mmap

Regular file I/O:
User Buffer → copy → Kernel Buffer → copy → Disk
(Two copies!)

mmap() file I/O:
User Virtual Address → Page Fault → Kernel loads page from disk → Direct access
(Zero copies! Pages loaded on demand)

🧠 THEORY: mmap() Parameters Explained

void *mmap(
    void *addr,         // Suggested address (usually NULL = let kernel choose)
    size_t length,      // Size to map
    int prot,           // Protection: PROT_READ | PROT_WRITE | PROT_EXEC
    int flags,          // MAP_SHARED | MAP_PRIVATE | MAP_ANONYMOUS
    int fd,             // File descriptor (-1 for anonymous)
    off_t offset        // Offset in file (must be page-aligned)
);
Flag Meaning
MAP_SHARED Changes visible to other processes, written to file
MAP_PRIVATE Copy-on-write, changes NOT written to file
MAP_ANONYMOUS Not backed by file (memory allocation)
MAP_FIXED Map at exact address (dangerous!)

// File: ~/embedded_learning/day_06/mmap_complete.c
#include <stdio.h>
#include <stdlib.h>
#include <fcntl.h>
#include <unistd.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <string.h>
#include <sys/wait.h>

// Pattern 1: Read file via mmap (zero-copy)
void mmap_read_file(const char *filename) {
    int fd = open(filename, O_RDONLY);
    if (fd < 0) { perror("open"); return; }

    struct stat st;
    fstat(fd, &st);

    char *data = mmap(NULL, st.st_size, PROT_READ, MAP_PRIVATE, fd, 0);
    if (data == MAP_FAILED) { perror("mmap"); close(fd); return; }

    // Access file like an array — NO read() calls needed!
    printf("File size: %ld, First 50 chars: %.50s\n", st.st_size, data);

    // Can search, parse, etc. using normal pointer operations
    // The kernel loads pages on-demand (page faults)

    munmap(data, st.st_size);
    close(fd);
}

// Pattern 2: Shared memory between processes
void mmap_shared_memory() {
    size_t size = 4096;

    // Anonymous shared mapping — persists across fork()
    int *shared = mmap(NULL, size, PROT_READ | PROT_WRITE,
                       MAP_SHARED | MAP_ANONYMOUS, -1, 0);
    if (shared == MAP_FAILED) { perror("mmap"); return; }

    shared[0] = 0;  // Counter
    shared[1] = 0;  // Flag

    pid_t pid = fork();
    if (pid == 0) {
        // Child increments counter
        for (int i = 0; i < 1000; i++) shared[0]++;
        shared[1] = 1;  // Signal done
        _exit(0);
    } else {
        wait(NULL);
        printf("Shared counter: %d (expect ~2000, race possible without lock)\n",
               shared[0]);
    }

    munmap(shared, size);
}

// Pattern 3: Memory-mapped hardware simulation
// In real embedded: mmap /dev/mem or driver provides mmap
void mmap_hardware_sim() {
    // Simulate hardware register access
    size_t size = 4096;
    volatile uint32_t *regs = mmap(NULL, size, PROT_READ | PROT_WRITE,
                                    MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
    if (regs == MAP_FAILED) { perror("mmap"); return; }

    // In real driver, this would be physical hardware address
    regs[0] = 0x01;  // Control register
    regs[1] = 0xFF;  // Status register
    printf("Simulated HW regs: CTRL=0x%X, STATUS=0x%X\n", regs[0], regs[1]);

    // IMPORTANT: volatile prevents compiler from optimizing away reads
    // Without volatile, compiler might cache the value

    munmap((void*)regs, size);
}

int main() {
    // Create test file
    int fd = open("test.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    const char *text = "Hello World! This is a test file for memory mapping.\n";
    write(fd, text, strlen(text));
    close(fd);

    printf("=== MMAP FILE READ ===\n");
    mmap_read_file("test.txt");

    printf("\n=== MMAP SHARED MEMORY ===\n");
    mmap_shared_memory();

    printf("\n=== MMAP HARDWARE SIM ===\n");
    mmap_hardware_sim();

    unlink("test.txt");
    return 0;
}

📝 INTERVIEW QUESTIONS — Day 6 (8 Questions)

Q1: mmap vs read/write — when to use each?

mmap: random access, multiple reads of same data, shared memory, large files, HW register access. read/write: sequential one-time access, small files, simple operations.

Q2: MAP_PRIVATE vs MAP_SHARED?

PRIVATE: COW — changes not written to file, not visible to others. SHARED: changes go to file and visible to all mapping processes.

Q3: What is the purpose of msync()?

Forces dirty pages to disk. Without it, changes may be lost on crash. msync(addr, len, MS_SYNC) blocks until write completes.

Q4: How does mmap relate to device drivers?

Drivers implement .mmap() in file_operations to map device memory/DMA buffers into user space. User calls mmap() on /dev/mydevice, kernel calls driver’s mmap handler, which sets up page table entries pointing to physical device memory.

Q5: Why use volatile with memory-mapped registers?

Without volatile, compiler may cache register values in CPU registers, reorder accesses, or eliminate “redundant” reads. volatile forces every access to go to the actual memory address.

Q6: What happens when you access unmapped mmap’d memory?

Page fault → kernel checks if address is in valid mapping → if yes, loads page from file/allocates zero page → returns to user. If address invalid → SIGSEGV.

Q7: Can mmap() fail? When?

Yes. Common failures: out of virtual address space, file too large, invalid flags, permission denied, no physical memory for page tables. Always check for MAP_FAILED.

Q8: What is the difference between mmap() and ioremap() in kernel?

mmap() is userspace syscall. ioremap() is kernel function that maps physical I/O addresses into kernel virtual address space for driver register access. Both achieve memory-mapped I/O but in different contexts.

📝 INTERVIEW QUESTIONS — Days 7-8 (12 Questions)

Q1: What is initramfs and why is it needed?

Temporary root filesystem in RAM. Contains drivers and tools needed to mount real root. Needed because root may be on complex storage (RAID, LVM, encrypted, NFS) requiring drivers not built into kernel.

Q2: Difference between /proc and /sys?

/proc: process info + kernel params, historical, somewhat messy format. /sys: structured device/driver info, modern, organized by bus/class, each file=one value. Use /sys for device driver interfaces.

Q3: What is PID 1 and why is it special?

Init process (systemd). First user process. Cannot be killed. Adopts orphans. Reaps zombies. System shuts down if PID 1 exits.

Q4: What is U-Boot?

Universal Boot Loader — standard bootloader for embedded Linux (ARM, MIPS, etc.). Initializes hardware (DRAM, clocks), loads kernel + DTB + rootfs from flash/SD/TFTP/USB, passes control to kernel with boot args.

Q5: What is the kernel command line?

Parameters passed to kernel at boot (by GRUB/U-Boot). Example: root=/dev/mmcblk0p2 console=ttyS0,115200 loglevel=7. Controls root device, console, debug settings.

Q6: What is the difference between insmod and modprobe?

insmod: loads a specific .ko file, no dependency handling. modprobe: searches /lib/modules/, resolves and loads dependencies automatically. Use modprobe in production, insmod during development.

Q7: How does udev create device nodes?

Kernel sends uevent when device registered. udev daemon (or mdev on embedded) receives event, creates /dev node with correct major/minor/permissions based on rules in /etc/udev/rules.d/.

Q8: What is sysfs and how do drivers use it?

Kernel object filesystem at /sys. Drivers expose attributes (readable/writable files) via DEVICE_ATTR_RO/RW/WO. Users read/write these files to control driver behavior without ioctl.

Q9: What is the difference between built-in and loadable modules?

Built-in (obj-y): compiled into kernel image, available at boot, __init memory freed after boot, cannot be removed. Loadable (obj-m): separate .ko file, loaded/unloaded at runtime, uses more memory.

Q10: Explain rootfs mounting process.

  1. Kernel mounts initramfs as temporary /. 2. initramfs /init loads storage drivers. 3. Mounts real root partition. 4. switch_root or pivot_root changes to real root. 5. Real /sbin/init starts.

Q11: What is BusyBox and why is it used in embedded?

Single binary providing hundreds of Unix utilities (sh, ls, cat, mount, etc.). Uses symlinks: /bin/ls → /bin/busybox. Tiny footprint (~1MB for everything). Standard for embedded rootfs.

Q12: How do you cross-compile a kernel for ARM?

bash export ARCH=arm export CROSS_COMPILE=arm-linux-gnueabihf- make defconfig # or vendor-specific config make menuconfig # customize make zImage dtbs modules -j$(nproc) Produces arch/arm/boot/zImage (kernel) and arch/arm/boot/dts/*.dtb (device trees).
